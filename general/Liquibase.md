GIT Based Deployment Strategy

1. How is idempotency achieved?
    - Everytime we execute Liquibase `update()` command
        - Entry recorded in DBChangelog table added for tracking purposes
        - For each changeset in changelog:
            - Look up on ID + AUTHOR + FILENAME in DATABASECHANGELOG table -> Identity
                - If not found -> execute and record it in DATABASECHANGELOG table
                - If found ->
                    - Checks MD5 sum of DDL/SQL -> Tamper Detection
                        - Matches - SKIP (Already run, unchanged all good)
                        - Differs - FAIL (Someone modified an already-executed changeset)
    - Post `update()` command execution we execute Liquibase `tag()` command

2. Rollbacks?
    - Need to specify rollback target `tag` version (`v`)
    - Validate if tag `v` actually exists
    - We identify all changesets after `v`
    - For each changeset after tag version (descending order), 
        - Liquibase executes rollback SQL (autogenerated if not specified) specified in changeset
        - Delete the row from DATABASECHANGELOG table     

3. Workflow
    - Create GIT repo for each database
    - Folders for each schema
    - In each schema create Liquibase yamls per DB object
    - DATABASECHANGELOG table created per DB, Schema combination
    - liquibase --update
    - liquibase --tag
